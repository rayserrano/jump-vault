Engineering

The stack is already chosen.

Jump named its data stack when it described this role: Snowflake for the warehouse, dbt for transformation, a semantic layer for governed access, and agents that consume what the platform serves. Nothing on this page proposes a tool. It takes the stack Jump already picked and shows it put to work: the decision behind each piece, the code that carries the patterns, and the guides a team runs on. The Architecture page said what the platform is. This page shows how it gets built and how it stays teachable.

The Stack, Put to Work

Each piece is a decision with a reason attached.

A tool list proves familiarity. A decision list proves judgment. Each entry below states what the piece does in this platform, why it earns the place, and what it costs, and each one ties back to a view on the Architecture page.

Snowflake holds every layer the Stack view drew, from landed source data to the club marts. It also enforces the ownership promise: each club’s mart is a separate space with its own access controls, so ownership is something the warehouse guarantees rather than something a diagram claims. The cost is shared compute that needs governing, and the platform budgets for that from the first day.

dbt is where the Universal Data Model integration layer gets built, tested, and versioned. Every transformation is code in one repository, reviewed before it merges and repeatable after, which is what turns onboarding a new club from custom work into a pattern the team stamps again.

The dbt Semantic Layer enforces what the Decisions view promised. A metric is defined once, in code, and every consumer receives that same definition at query time. Self-serve stops drifting because there is nothing local left to drift.

Agents and language models reach the platform through the Model Context Protocol, the open standard that connects them to the dbt Semantic Layer. The intelligence layer asks the same governed definitions a dashboard uses, so an agent’s answer and an analyst’s answer agree by construction. And the stack is moving the same direction: dbt’s first launches after its June 2026 merger with Fivetran included Agents Schema, an open standard that publishes metric definitions, semantic models, and lineage into a governed schema in the warehouse as shared context for agents. Betting on governed agent access is betting with the stack’s own roadmap.

Ingestion follows one buying rule: use an off-the-shelf connector where a good one exists, build only where the source demands it, and hold both to the same tests and monitoring. The default is Fivetran, for the depth of its managed connector library, and because it is now one company with dbt, which puts the platform’s loading and transformation layers on a single roadmap.

Sports produces sources no catalog covers: a club’s legacy system, a league feed that arrives as files, a venue system that speaks only SFTP. Those get a custom loader, and the discipline is that a custom loader earns no exceptions. It lands as history like every other source, carries the same namespaced keys, and passes the same tests and monitoring, so from the integration layer inward, hand-built and off-the-shelf are indistinguishable. The loader pattern itself appears below.

Version control, code review, CI/CD, automated testing, lineage, and observability run under all of it. They are what let the platform take on new clients and new sources without growing brittle.

The Patterns

Working code carries the claims.

Everything above is easy to say. So this section shows seven exhibits from the pattern library, written fresh for this study in Jump’s own domain of fans, clubs, tickets, and orders. Every table, name, and number in them is illustrative. The first four put the founding decision into running code: a relationship is the join of two party roles, and tenancy lives on the relationship, never on the fan. The fifth shows what building a loader looks like when no connector exists, the sixth shows the storage pattern both loading paths land in, and the seventh shows the semantic definitions every consumer, human or agent, receives.

models/integration/ticketing/integrator_ticketing_fan_club_relationship.sql
-- Ticketing memberships -> PARTY_RELATIONSHIP.
-- Links a fan (FAN role) to a club (CLUB role), one row per fan-club edge.
-- Tenancy lives here, on the relationship, never on the fan: the same
-- person holding memberships at two clubs produces two edges and stays
-- one PARTY. Rows are only built where both sides already resolved.

{{
    config(
        materialized="merge_into",
        meta={
            "target_schema": "INTEGRATION",
            "target_table": "PARTY_RELATIONSHIP",
            "unique_key": "PARTY_RELATIONSHIP_ID",
            "diff_key": "PARTY_RELATIONSHIP_HDIFF",
            "ts_key": "UPDATED_AT",
        },
    )
}}

WITH source AS (
    {{
        current_from_history(
            history_rel=ref("hist_ticketing_membership"),
            key_column="membership_hkey"
        )
    }}
),

resolved AS (
    SELECT
        fan.party_code  AS fan_party_code,
        club.party_code AS club_party_code,
        s.member_since,
        s.load_ts_utc
    FROM source s
    INNER JOIN {{ ref('party_read') }} fan
        ON fan.party_code = 'TIX|FAN|' || CAST(s.fan_id AS VARCHAR)
    INNER JOIN {{ ref('person_read') }} per
        ON per.party_id = fan.party_id
    INNER JOIN {{ ref('party_read') }} club
        ON club.party_code = 'TIX|CLUB|' || CAST(s.club_id AS VARCHAR)
    INNER JOIN {{ ref('organization_read') }} org
        ON org.party_id = club.party_id
),

final AS (
    SELECT
        {{ dbt_utils.generate_surrogate_key(
            ["fan_party_code", "'FAN'", "club_party_code", "'CLUB'"]
        ) }} AS party_relationship_id,
        {{ dbt_utils.generate_surrogate_key(
            ["fan_party_code", "'FAN'"]
        ) }} AS from_party_role_id,
        {{ dbt_utils.generate_surrogate_key(
            ["club_party_code", "'CLUB'"]
        ) }} AS to_party_role_id,
        member_since AS from_date,
        CAST(NULL AS DATE) AS thru_date,
        {{ dbt_utils.generate_surrogate_key(
            ["fan_party_code", "club_party_code", "member_since"]
        ) }} AS party_relationship_hdiff,
        load_ts_utc AS updated_at
    FROM resolved
)

SELECT * FROM final

The walk from top to bottom. The source reads current membership state from history. Resolution joins each side to the party layer through namespaced source keys, and the subtype joins prove the fan is a person and the club is an organization before any edge is built. The surrogate keys then compose the two party roles, fan on one end and club on the other, into one relationship row. When a second club arrives, nothing about the fan changes. The fan stays one party, and the platform adds an edge.

The test file mirrors production discipline: one logic area, named paths, mocked inputs, explicit expectations. Path one proves the edge is built when both sides resolve. Path two proves a missing club produces no orphan. Path three is the one that matters: one fan, memberships at two clubs, and the expectation is two relationship rows and still one person. Here the founding decision has an enforcer. If tenancy ever drifts onto the fan, this test fails and the build stops.

The third exhibit is where dbt gets extended rather than only used. dbt ships tables, views, and incremental models. A platform with one canonical relationship table and many sources writing into it earns its own materialization, and every writer declares the same contract:

models/integration/merch/integrator_merch_customer_club_relationship.sql
-- A second source, the merch store, writing to the same canonical table.
-- Different source keys, identical contract, identical guarantees.

{{
    config(
        materialized="merge_into",
        meta={
            "target_schema": "INTEGRATION",
            "target_table": "PARTY_RELATIONSHIP",
            "unique_key": "PARTY_RELATIONSHIP_ID",
            "diff_key": "PARTY_RELATIONSHIP_HDIFF",
            "ts_key": "UPDATED_AT",
        },
    )
}}

The materialization behind that contract makes three guarantees. Each run merges on the unique key, so writers are idempotent and reruns are safe. The hash diff means unchanged rows are never rewritten, which keeps warehouse cost proportional to real change. The timestamp key means late-arriving data can never regress newer state. Ticketing, merch, and every source after them inherit all three by declaring the same few lines, and that is how one table stays canonical while the source count grows.

And this is the machinery those guarantees come from:

macros/materializations/merge_into.sql (the custom materialization)
-- Custom materialization: merge_into
-- dbt assumes one model builds one table. The integration
-- layer needs the opposite: many source models writing one
-- canonical table. This materialization is the bridge. The
-- model's SELECT is the transformation; the MERGE is the
-- load; there is no buffer table between them.
-- Production adds an exclude-from-update list to protect
-- audit columns; the shape is what matters.

{% materialization merge_into, adapter='snowflake' %}

    {%- set meta = config.get('meta', {}) -%}
    {%- set target_schema = meta.get('target_schema', target.schema) -%}
    {%- set target_table = meta.get('target_table') -%}
    {%- set unique_key = meta.get('unique_key') -%}
    {%- set diff_key = meta.get('diff_key') -%}
    {%- set ts_key = meta.get('ts_key', 'UPDATED_AT') -%}

    {% if not target_table or not unique_key %}
        {{ exceptions.raise_compiler_error(
            "merge_into requires meta.target_table and meta.unique_key"
        ) }}
    {% endif %}

    -- The canonical table is DDL-owned, not dbt-managed.
    {%- set target_relation = api.Relation.create(
        database=target.database,
        schema=target_schema,
        identifier=target_table
    ) -%}

    -- Build the update list from the target's own columns.
    {%- set columns = adapter.get_columns_in_relation(target_relation) -%}
    {%- set column_names = columns | map(attribute='name') | list -%}

    {%- set merge_sql -%}
        MERGE INTO {{ target_relation }} AS tgt
        USING ( {{ sql }} ) AS src
        ON tgt.{{ unique_key }} = src.{{ unique_key }}

        WHEN MATCHED
            AND tgt.{{ diff_key }} != src.{{ diff_key }}
            AND src.{{ ts_key }} >= tgt.{{ ts_key }}
        THEN UPDATE SET
            {% for col in column_names if col != unique_key %}
            tgt.{{ col }} = src.{{ col }}{% if not loop.last %},{% endif %}
            {% endfor %}

        WHEN NOT MATCHED THEN INSERT (
            {% for col in column_names %}
            {{ col }}{% if not loop.last %},{% endif %}
            {% endfor %}
        )
        VALUES (
            {% for col in column_names %}
            src.{{ col }}{% if not loop.last %},{% endif %}
            {% endfor %}
        )
    {%- endset -%}

    {%- call statement('main') -%}
        {{ merge_sql }}
    {%- endcall -%}

    -- The model owns no relation of its own; the canonical
    -- table is the product.
    {{ return({'relations': []}) }}

{% endmaterialization %}

A materialization is the deepest extension point dbt offers, and this one bridges the framework’s one-model-one-table assumption to the integration layer’s needs. The model’s SELECT is the transformation, the MERGE is the load, and there is no buffer table between them. The materialization refuses to compile without the key contract, builds its update list from the target table’s own columns, and applies the hash diff and newer-wins rules inside the MERGE itself, so no writer can opt out of them. Every integrator that declares a few lines of config inherits the same implementation of merge correctness.

macros/landing/ingest_into_landing.sql (custom loader for a file-drop source)
-- The league schedule arrives as gzipped TSV files on S3,
-- date-stamped, with no connector in any catalog. The loader
-- is still dbt code: the stage, the file format, and the
-- landing schema are themselves created by macros in this
-- same repository, reviewed and run like every other model.

{% macro ingest_into_landing_sql(
        full_table_name, field_count,
        file_pattern, full_stage_name, full_format_name
) %}
BEGIN TRANSACTION;

COPY INTO {{ full_table_name }}
FROM (
    SELECT
        {% for i in range(1, field_count+1) %}${{i}},{% endfor %}

        METADATA$FILENAME        AS from_file,
        METADATA$FILE_ROW_NUMBER AS file_row_number,
        '{{ run_started_at }}'   AS ingestion_ts_utc

    FROM @{{ full_stage_name }}/
)
PATTERN = '{{ file_pattern }}'
, FILE_FORMAT = (FORMAT_NAME = '{{ full_format_name }}')
;

COMMIT;
{%- endmacro %}

-- Every landed row carries its provenance: the file it came
-- from, the row of that file, and the moment it arrived. A
-- support question about one game in one schedule reaches
-- the source file in one hop.

When Fivetran covers a source, the platform buys the connector. When nothing does, this is what building looks like: the loader’s whole infrastructure is code in the same repository, file selection is a date-windowed pattern, and every landed row knows exactly where it came from. The two paths converge at landing, and everything downstream treats them identically.

macros/storage/save_history.sql + current_from_history.sql (the storage pair)
-- Every source lands as insert-only history. Nothing a
-- source ever said is updated or deleted; new versions are
-- appended, and the anti-join below means history grows
-- only by real change. Current state is not a table anyone
-- maintains. It is one window function over history.
-- Production adds filter and high-watermark options; the
-- shape is what matters.

{% macro save_history(
    input_rel, key_column, diff_column,
    load_ts_column="LOAD_TS_UTC"
) -%}

    {{ config(materialized="incremental") }}

    with

        {%- if is_incremental() %}
            current_from_history as (
                {{ current_from_history(
                    history_rel=this,
                    key_column=key_column,
                    selection_expr=diff_column
                ) }}
            ),

            load_from_input as (
                select i.*
                from {{ input_rel }} as i
                left outer join current_from_history as h
                    on h.{{ diff_column }} = i.{{ diff_column }}
                where h.{{ diff_column }} is null
            )

        {%- else %}
            load_from_input as (
                select * from {{ input_rel }}
            )
        {%- endif %}

    select * from load_from_input

{%- endmacro %}

{% macro current_from_history(
    history_rel, key_column,
    selection_expr="*", load_ts_column="LOAD_TS_UTC"
) -%}

    select {{ selection_expr }}
    from {{ history_rel }}
    qualify
        row_number() over (
            partition by {{ key_column }}
            order by {{ load_ts_column }} desc
        ) = 1

{%- endmacro %}

The pair closes a loop the first exhibit opened: the relationship integrator’s source CTE called current_from_history, and this is what it called. save_history is self-referential, asking its own history what it already knows and inserting only versions whose hash diff is new. The guarantees fall out of the shape. Replay is possible because nothing is ever lost. Audit is possible because every version a source sent is still there. And current state cannot drift from history because it is derived from it, one window function, with no update logic to get wrong.

models/mart/semantic_membership_fact.yml (the semantic layer, and the agent’s briefing)
# The descriptions below are written for two readers: the
# engineer who maintains this file, and the agent that
# serves it. An agent that can quote a definition can
# explain it.

version: 2

semantic_models:
  - name: membership_fact
    description: >
      Fan-club membership fact. Grain: one row per fan-club
      membership edge, straight from PARTY_RELATIONSHIP.
      Illustrative population: ~1.6M edges across clubs.
      Coverage note: memberships whose club has not resolved
      in the integration layer are excluded upstream by the
      integrator's inner joins. The no-orphan unit test in
      the second exhibit is what enforces that boundary.
    model: ref('membership_fact')
    entities:
      - name: membership
        type: primary
        expr: party_relationship_id
      - name: fan
        type: foreign
        expr: from_party_role_id
      - name: club
        type: foreign
        expr: to_party_role_id
    dimensions:
      - name: member_since
        type: time
        type_params:
          time_granularity: day
      - name: thru_date
        type: time
        type_params:
          time_granularity: day
    measures:
      - name: membership_count
        description: Base additive measure. One per edge.
        agg: sum
        expr: "1"

metrics:
  - name: active_member_count
    description: >
      Memberships with no thru_date. The number every club
      dashboard and every agent answer must agree on.
    type: simple
    type_params:
      measure: membership_count
    filter: |
      {{ Dimension('membership__thru_date') }} is null

  # renewal_count and lapsed_count: simple filtered metrics,
  # same shape as above, omitted for length.

  - name: renewal_rate
    description: >
      Renewals over lapsed memberships. The denominator is
      lapsed, never total: totals mix active edges into the
      base and make the rate look better than it is. The
      denominator and its reason live here, in the
      definition, so no consumer can pick a different one
      and any agent asked why can answer from this text.
    type: ratio
    type_params:
      numerator: renewal_count
      denominator: lapsed_count

The descriptions are the point, and they are written for two readers. For the engineer, they carry the governance: the grain, the coverage boundary, and the denominator with its reason, sitting where no consumer can pick a different one. For the intelligence layer, they are the briefing: this file is what the platform serves through MCP, so when someone asks an agent for the renewal rate, the agent answers from this definition, and when they ask why lapsed is the denominator, the answer is already in the metadata it read. The guides below teach the human team. This file is the same enablement aimed at the platform’s other consumer.

Three more patterns run underneath these exhibits and get named rather than shown. Ephemeral read markers declare dependencies on shared tables, so build order is enforced and unit tests can mock cleanly. A verification layer reconciles each build against a captured baseline before the build counts. And a normalization macro family standardizes messy source values in one governed place. Each is a pattern the team inherits, and the guides below teach them.

The Enablement

The platform is finished when the team can run it without its author.

A strong engineer leaves behind working systems. A platform leader leaves behind a team that can extend them. This section holds implementation guides written for the engineers who will live in the stack, and every guide follows the same shape: prerequisites, the steps with a worked example beside each one, the reason behind each step, validation checks with expected results, and the mistakes to expect. The shape is the point. A guide built this way keeps the platform teachable rather than tribal, and it is how standards survive a growing team.

Three parts of the skeleton carry most of the weight. guides/onboarding_a_new_club_mart.md (excerpts)
## 1.3 What is intentionally not done

This guide does not touch identity resolution. The mart
consumes resolved parties from the integration layer; it
never creates or merges them. If onboarding surfaces fans
the platform cannot resolve, that is integration-layer work
and it has its own guide. Doing it here would put club-scoped
logic inside the shared layer, and the whole tenancy model
depends on that line staying where it is.

## 6.3 Validation check three: relationship edge count

SELECT COUNT(*) AS edge_count
FROM INTEGRATION.PARTY_RELATIONSHIP pr
INNER JOIN INTEGRATION.PARTY_ROLE r
    ON pr.to_party_role_id = r.party_role_id
INNER JOIN INTEGRATION.PARTY p
    ON r.party_id = p.party_id
WHERE p.party_code = 'TIX|CLUB|77';

Expected: within two percent of the club's source
membership count (illustrative: 41,900 to 42,700 against a
42,310-row extract). A count far below means resolution is
dropping fans. A count above means the uniqueness test in
section 4.1 has a gap. Either way, stop before the mart
build and find out which.

The version line at the top is the part that matters most. Guides ship as v1, and the implementing engineer audits them before executing. The audit comes back as findings, and v2 answers every finding by name in its own header. Open questions get the same routing, split between what is already resolved and what the engineer can answer from the repo and the source data. This is practiced rather than invented: the delivery history behind this page carries exactly these headers. A guide the engineer can audit and change is the difference between a team that executes instructions and a team that owns the platform, and the version history is the receipt.
