Architecture

Eight layers hold the whole platform.

The rest of the site shows the architecture working. This page names it, one layer at a time, from the foundation to the open frontier. Every choice here is a settled pattern with its reasons on the table, the forward-leaning layers included.

One platform, read three ways.

An architecture is never one picture. This one is read from three angles: the layers it is built from, the decisions it exists to serve, and the ways it is consumed. Each view answers a different question, and together they are the whole platform.

The Map shows the platform at a glance. This goes a level deeper: the layers themselves, every source resolving through one governed integration layer that all the tools above it inherit. Tap any layer to see what happens inside it.

View 2 · The Decisions

How value comes out: one governed truth, every path ending in a decision.

The stack exists to be consumed. This view spreads the top of it out: every consumer reaches the same governed source, through the tool that fits them, and each path ends not in a dashboard but in a decision. The one split that matters is whether the metric is defined once and governed, or improvised per team.

The distinction that carries this view is governance. Self-serve tools let each team define its own metrics, and those definitions drift until two dashboards disagree. The semantic layer defines each metric once and serves it everywhere, so an executive, an analyst, and an agent are all standing on the same number. Every consumer ends in a decision rather than a report, and because the ground under them is one governed truth, the decisions are consistent by construction.

View 3 · The Consumption Layer

One governed truth, consumed three ways.

The serving layer meets three kinds of demand from one governed source. Marts answer classic analytics, the questions that look back and across: how did the season go. An operational store answers live counts in the moment, the number a floor manager needs before kickoff, the reporting the transactional apps rarely do well. And interactive analytics answers at volume: fan-facing analytics served to thousands at once, sub-second, and where the workload fits it keeps serving inside the governed platform rather than a separate engine. Increasingly, none of these is read by a person alone. Every consumer now arrives with an agent, and the same governed truth serves them both.

Platform Principles

Eight principles the platform is built on.

The three views show what the platform is and how it is consumed. These are the decisions underneath it: the reasons it is built this way, each one a position taken against a worse alternative.

The Foundation

One layer absorbs the complexity; the rest inherit simplicity

One integration layer beats fifty stitched marts.

This is Inmon’s Corporate Information Factory: one integration layer holds the canonical fan, and client marts are derived above it. The alternative, fifty club warehouses stitched together after the fact, is how two-truths disease starts. The integration-first approach is the correct one, and the one teams most often avoid, because it front-loads the hard modeling work before anything ships. That calculus is what the agentic era changes. The work that made the integration layer too expensive to build by hand is exactly the work an agent now absorbs, so the reason to skip it no longer holds.

The Model

Change arrives as data, not schema

Most models die of sprawl. This one does not.

First, what this is not: nobody consumes this layer. Fans, clubs, and analysts read the club marts and the semantic layer above it. This is the foundation they stand on, kept rigorous here precisely so the surface they touch stays simple.

Most data models in the industry face the same slow death. A new business concept arrives, someone alters a table, and the model grows a column; a few years of that and an entity carries hundreds of flag columns, most of them storing the word no. Sprawl compounds, every change becomes a migration, and the platform calcifies until replacing it is cheaper than changing it. This is the ordinary way data platforms die.

The Universal Data Model is the compression of that whole problem. The model here is Silverston’s, and the pedigree runs decades deep. The macro pattern is a conglomeration of proven patterns, composed once. Classification is a pattern, a set of tables. Status is a pattern, a set of tables. So is item, so is role. Onboarding a new entity is not building tables; it is applying those patterns, which already exist. Each application leaves only a thin per-entity trace, and everything else it needs arrives as rows in structures that are already built.

One pattern makes the whole idea concrete. Take classification, the pattern that answers what a fan is. The sprawling model on the left grows a column for every label, which is why it drowns in flags storing no. The pattern holds the labels as rows instead, so a fan can carry any number of them and a brand-new label is a single insert. The season a club invents a new tier, no table changes.

The upside is a platform that does not compound its maintenance cost as it grows. New requirements become inserts instead of migrations. The same pattern absorbs permutations no one designed for: many labels at once, labels that change over time, whole new dimensions of them. Requirements shift the way business always shifts, and the model meets them with rows rather than a maintenance project, so the cost that normally compounds with every change never accrues. Club twelve onboards exactly the way club three did. The marts, the semantic layer, and the agents all stand on a model that does not shift under them. The pattern library and the rules for composing it are precisely what the implementation guides on the Mentoring page teach.

The macro pattern is best seen as a team of collaborating patterns, each one a small, proven solution to a recurring modeling problem: how to classify a thing, how to track where it is in its life, how it takes part, what its pieces are. The patterns already exist and already work together. Onboarding a new entity is not building it from parts; it is handing it to a team that is standing by.

The Access

One definition, governed once, served everywhere

The semantic layer is the constitution.

Every consumer on the data platform, human or agent, reads metrics from one governed semantic layer. A metric is defined once, in the open, with lineage, and no dashboard, model, or rule may redefine it. This is what makes agent-authored artifacts safe: a dashboard built on governed metrics cannot lie about revenue.

The Intelligence

Give each kind of intelligence the job it does best

Models predict. Agents author. Rules execute.

The division of labor is strict. ML models predict: propensity, uplift, and the scores on every fan row. Agents author: they read the warehouse, write rules, build assets, and retire what stops working. Rules execute: every high-volume decision runs deterministically. A rule is predefined if-this-then-that logic a machine runs the same way every time. Nothing with a per-transaction cost sits inside a language model loop.

The Serving

Match the serving engine to the workload

Analytics at scale is a new paradigm, and the warehouse grew an arm to meet it.

The query load changed shape. Club analysts once ran bursty dashboards where a few seconds were fine. Now embedded analytics for every club, fan-facing surfaces, and agents in reasoning loops produce thousands of small queries around the clock, where a slow answer stalls an agent mid-reasoning. This workload wants cost-per-query economics, not the warehouse-seconds the core is priced in.

It computes, in seconds.
SERVING the new arm
High-concurrency OLAP for surfaces and agents
It answers, in milliseconds.

So the warehouse grew an arm for it: a serving tier of high-concurrency, millisecond answers, materializing governed metrics from the semantic layer, with data flowing one way: integration, then marts, then materialize, then serve. A serving copy makes freshness and consistency governed properties too; ignore that and you reinvent the two-truths disease the platform exists to cure.

Everywhere else on this site the vendors are settled. Here they are not, and deliberately so. The arm the warehouse grew is Snowflake’s Interactive Analytics, serving inside the governed platform; the alternative is a dedicated serving engine, a specialist faster at the extremes. Which one is a criteria decision: concurrency, latency, cost per query, operations burden, and semantic-layer integration. The warehouse vendors moved into interactive serving this year, and the dedicated engines are not standing still. The architecture holds either way, and the choice is margin engineering: platform cost per client lands directly on the Path to Black.

The Execution

Decide in the cheap engine, not the expensive one

The rules engine decides at zero token cost.

Rules engines are the right way to make high-volume decisions, and the one constraint that ever held them back was the human labor of maintaining rules at scale. Thousands of rules are more than a team can keep current by hand, and that maintenance cost, not any flaw in the approach, kept rules engines from their promise. That constraint is what the agentic era removes: an agent observes outcomes through the warehouse, retires stale rules, proposes refinements, and backtests before anything deploys, so the labor that limited the engine is the labor the agent now carries.

This is not unique to rules engines. It is a pattern. The integration-first data platform on this page is the same story: correct, and set aside because the modeling work came before anything shipped. d3, the most expressive visualization library there is, is the same story again: teams reach for an off-the-shelf tool like Tableau because d3 asks for hand-work they cannot spend. Each is a superior capability the field abandoned under one accepted wisdom, that it costs too much by hand. The agentic era voids that wisdom, because the hand-work is exactly what an agent now absorbs.

The advice follows: resist surrendering to accepted wisdom the new era has made obsolete. The advantage goes to whoever builds the superior capability while the field still believes it is too expensive. Most teams will keep reaching for the convenient option, on priors that were true a year ago and are not true now. That gap is the opening.

And rules fix what stalls agents, because tokens cannot price a per-transaction decision, while a compiled rule decides at a marginal cost near zero. AI makes rules possible. Rules make AI possible at scale.

The loop is the flywheel’s inner gearbox: the agent analyzes the warehouse, proposes a rule, the backtest scores it, a human approves inside guardrails, the engine deploys, telemetry flows back, and the agent refines. Every decision traces to a rule version: inspectable, reversible, and auditable, which matters more the day the first European club goes live and automated decisions meet European regulation.

Jump’s own launch copy describes the Pricing Manager modeling scenarios and pushing pricing changes across events in minutes. That is rule authorship and deployment, already shipped. Five of the nine applications are rules domains wearing product clothes: pricing, yield, journey triggers, loyalty accrual, and ingress entitlements. Vault proposes one governed decision layer instead of five embedded ones. The general law follows: the agent is the composer and the analytics estate is the composition. Models, rules, and BI assets are all authored expensively in tokens, executed at near-zero cost, and governed by the same semantic constitution.

The Boundary

Centralize the integration, federate the consumption

One truth in the middle, many copies at the edge.

Data mesh is an ownership discipline, not a decentralization of truth: each domain owns its data and publishes it as a product. Read as decentralization, mesh looks like the rival of one central truth. It is not. Centralization and mesh stop being rivals when each takes the half it is good at. The integration stays centralized: one canonical fan, one place identity resolves. The consumption federates: capable organizations receive governed data products, with contracts and lineage attached, published into their own stacks. That is the third DWaaS tier from the Map, and it is the boundary where the platform ends on purpose: a mesh of governed products stamped from one model, a Universal Data Mesh, if you will.

The Identity

Resolve the person once, own the relationship locally

The whole person lives in the platform. Each club owns its relationship.

The integration layer holds the resolved party: every source key mapped, every relationship typed, one record for the person no matter how many systems she arrived from. Each club’s mart carries only that club’s relationship with her. Progressive profiling, customer 360, and lifetime value are views stamped from the same resolved party, which is why they agree with each other, and why the fan graph option on the Bench page costs nothing to keep open.

Tenancy lives on the relationship, not on the person, so the model never assumes what kind of tenant it serves. A club and a touring act look the same to it: fans, transactions, and a lifecycle to own. The UDM would handle music fans exactly as it handles sports fans, same resolved person, same governed marts, for whenever someone decides to make them Jump.

The shared person is also an option on the future. Because every club reads the same resolved party, the platform can recognize when non-competing clubs, a basketball club and a football club in one market, are serving the same supporter. Each still owns its relationship with her. What a network of clubs chooses to build on that recognition, under governance, is the opportunity that sits above the platform.
